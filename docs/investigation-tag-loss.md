# 調査メモ: 種目に紐づいたタグが消失する事象(#8)

- ステータス: **調査のみ**(コード変更なし)
- 対象報告: 「種目に紐づいたタグ(高重量日など)が、稀に消えて タグなし に戻る」。再現手順なし。
- 関連: 暗転バグ #1 と併発している可能性がユーザーから指摘されている。

## 結論(先に要点)

書き込み経路のコードを精査した結果、**タグを決定的に消すバグは見つからなかった**。
タグ(`tagId`)を書き換える/巻き込む処理はすべて、対象セットを
トランザクション内で読み直し、`{ ...s, 変更フィールド }` の形で **既存フィールドを保持したまま**
`bulkPut` している。つまり単独の操作でタグだけが `NO_TAG('')` に化ける経路は無い。

したがって最有力仮説は **#1 との併発による「保存未完了」**、
次点で **fire-and-forget 書き込みのトランザクション暗黙アボートによる無言の保存失敗** である。

## 精査した書き込み経路(いずれも tagId を保持)

| 関数                                    | 挙動                                    | tagId の扱い                    |
| --------------------------------------- | --------------------------------------- | ------------------------------- |
| `changeBlockTag`                        | 対象ブロック全セットの tagId を付け替え | 明示的に `toTagId` へ(意図通り) |
| `changeBlockExercise`(#2 で追加)        | 対象ブロックの exerciseId を付け替え    | `...s` で保持(変更しない)       |
| `reorderSetsInBlock` / `moveBlockInDay` | 日全体を再採番して `bulkPut`            | `...s` で保持                   |
| `toggleSetAttribute` / `updateSet`      | `db.sets.update(id, 差分)`              | 差分に含めない → 保持           |
| `copyPreviousSession` / `transferSets`  | 履歴/別日から複製                       | 元セットの tagId を明示コピー   |

いずれも「読み → 変更 → 書き」をトランザクション内で完結させ、スナップショットも
トランザクション内で取り直しているため、**別トランザクションによるロストアップデートも起きにくい**。
UI 由来のスナップショット(`orderedIds` やブロックキー)が古くても、対象ブロックが
見つからなければ no-op で終わる(タグを壊さない)。

## 有力仮説

### 仮説A(最有力): #1 暗転により保存が完了していない

タグ付け操作(`changeBlockTag` 等)を行った直後〜前後で画面が暗転(#1)し、
書き込みが commit される前に操作が中断された場合、リロード後には
「タグを付けたのに消えている」ように見える。実際にはそもそも保存されていない。

→ #1 の対策(ErrorBoundary で復帰 UI 化 + sticky ヘッダーの `backdrop-blur` 除去による
iOS 合成バグ回避)を今回入れたため、これが原因なら発生確率は下がるはず。

### 仮説B(次点): fire-and-forget 書き込みの無言アボート

タグ変更は UI から `void changeBlockTag(...)` の形で発火しており、
Promise の reject を誰も拾っていない。iOS Safari で IndexedDB トランザクションが
まれに `abort`(バックグラウンド遷移・ストレージ圧迫・別タブ競合など)した場合、
**ユーザーには成功したように見えて実際は保存されていない**。

→ 推奨(今回未実施・調査のみのため): タグ/ブロック系の書き込み呼び出しに
`.catch()` を付け、失敗時に `showToast('保存に失敗しました')` で可視化する。
無言の消失を「保存失敗の明示」に変えられ、再現報告の切り分けにも役立つ。

## 却下した仮説

- **タグマスタの削除**: `deleteTag` は使用中タグの削除をブロックするため、
  記録側から tagId の参照先が消えることはない。
- **NO_TAG 正規化のバグ**: `tagId` は常に文字列(タグなしは `''`)で、
  `bulkPut` の `...s` で保持されるため、途中で `undefined` に化ける箇所は無かった。
- **マイグレーションでの欠落**: 現行スキーマ(v6)まで tagId を落とす変換は無い。

## 次アクション(推奨)

1. #1 の対策リリース後、事象が続くか観察する(仮説A の切り分け)。
2. 続く場合は仮説B 対応として書き込みへのエラー可視化(`.catch` + toast)を入れ、
   「保存失敗」を捕まえられるようにする。
3. それでも起きるなら、操作ログ(どのブロックにいつ何を書いたか)を
   一時的に残して実機で再現データを採取する。
